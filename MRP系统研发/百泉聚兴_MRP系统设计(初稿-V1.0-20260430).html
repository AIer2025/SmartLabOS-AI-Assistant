1. 系统概述与目标

1.1 业务背景

百泉聚兴交付给客户的智慧实验室是软硬件一体化的整体方案，以"产线"为最小交付单元，每条产线由 1～N 个平台（PLT-1200 / 1400 / 1800 等）拼装而成，每个平台搭载若干 SmartLabOS 模组（已规划 ~150 余种），模组又由若干层 BOM 物料组成。客户的项目落地节奏，强依赖于"每个模组在装配日期前能否齐套到位"。

1.2 目标

  • 项目级齐套可视化：以"客户—产线—平台—模组—BOM 物料"五层结构展示并跟踪交付进度。
  • 自动化 MRP 运算：依据装配日期反推，计算各模组/物料的毛需求 → 净需求 → 建议采购日。
  • 双轨齐套策略：
    • 成品模组：库存优先、不足部分按 BOM 反向爆炸下达采购；
    • 研发模组：URD → 研发计划 → 验证收敛 → 升级为成品模组并入库。
  • 数据可追溯：每一个 BOM 物料的状态、批次、领用记录、缺料预警，可追溯到具体客户产线。

1.3 范围（v1.0 MVP）

5.3 MRP 计算引擎（核心算法）

引擎入口：IMrpEngine.RunAsync(MrpInput)。计算分四步：

  1. 展开顶层需求：按 (产线 → 平台 → 模组实例) 汇总每个 ModuleCode 的需求台数 D_module[m]。
  2. BOM 反向爆炸：对每个模组，递归（一阶 / 多阶）展开 module_bom，得到每个 item_code 的毛需求 G[i] = Σ qty_per × D_module[m]。
  3. 抵扣库存与在途：净需求 N[i] = max(0, G[i] - 可用库存 - 在途采购) + 安全库存。
  4. 反推下单日：order_date = required_by_date − lead_time_days − safety_buffer；若 order_date < today，置 late_flag。
// C# 伪代码
public class MrpEngine : IMrpEngine
{
    public async Task<MrpResult> RunAsync(MrpInput inp, CancellationToken ct)
    {
        var moduleDemand = await _projectRepo.AggregateModuleDemandAsync(inp.ProjectCode);
        var bomMap       = await _bomRepo.LoadBomAsync(moduleDemand.Keys);
        var grossByItem  = BomExploder.Explode(moduleDemand, bomMap);

        var stock     = await _invRepo.GetOnHandAsync(grossByItem.Keys);
        var onOrder   = await _poRepo.GetOnOrderAsync(grossByItem.Keys);
        var itemMaster = await _itemRepo.LoadAsync(grossByItem.Keys);

        var reqs = new List<MrpRequirement>();
        foreach (var (item, gross) in grossByItem)
        {
            var meta = itemMaster[item];
            var need = Math.Max(0, gross - stock[item] - onOrder[item]) + meta.SafetyStock;
            var orderDate = inp.RequiredByDate.AddDays(-(meta.LeadTimeDays + inp.SafetyBufferDays));
            reqs.Add(new MrpRequirement {
                ItemCode = item, GrossQty = gross, NetQty = need,
                SuggestedOrderDate = orderDate,
                ShortageFlag = need > 0,
                LateFlag = orderDate < DateTime.Today
            });
        }
        await _runRepo.SaveSnapshotAsync(inp, reqs, ct);
        return new MrpResult(reqs);
    }
}
关键设计点： MRP 运算始终生成快照而不直接修改业务数据；采购、领料的真实变更走独立用例（保持"建议 vs 决策"分离）。这让用户敢于频繁重算 MRP，且历史可追溯。

9. 待讨论清单（与负责人确认后定稿）

  1. 多层 BOM：是否在 v1 就支持模组→子组件→零件的多阶 BOM？还是先按一阶展开 + 子组件用"虚拟模组"过渡？
  2. 项目实例 vs 主数据：当某条产线里"临时改了一颗物料的用量"，是否允许覆盖主数据？建议按项目级 BOM 覆盖（override 表）处理。
  3. 研发模组的 BOM 提前介入：URD 阶段是否允许预留 BOM？我们目前假设是研发完成后再录入 BOM。
  4. 采购批量策略：是否要支持最小起订量（MOQ）和包装规格（按 100 件起订）？这会影响 MRP 净需求向上取整。
  5. 跨项目共用件调拨：库存可能被多个项目预占，是否在 v1 就支持项目预留 vs 总库存分层（推荐支持）？
  6. 权限模型：项目经理只能看到自己负责的客户/项目，还是全公司可见？
  7. 与现有 Excel 流程并行期：是否需要导入/导出 Excel 同步给采购线下流转？建议支持《03-项目集成和交付清单》模板的双向导入导出。
  8. 价格字段：物料单价是否纳入系统？根据 SOUL.md 规则，价格信息由销售团队统一处理；MRP 默认不显示金额，仅在管理员视图开启。
  9. 研发→成品的编码策略：研发模组试制时使用 HHMxxx-UD 后缀（如 HHM10B-025-UD），转产时是否自动派生成新编码并保留映射？
  10. 通知渠道：缺料/迟料告警是否只走站内信，还是默认对接企业微信群机器人？

本设计文档将持续迭代。下一步建议在标注 "待讨论" 的 10 个问题上与项目负责人逐项确认，定稿后切入 M1 阶段。
