一、总体架构与双提案分型

公司当前阶段主营 SmartLabOS — 样品前处理阶段的软硬件一体化系统。售前提案的本质是把 "客户需求 + 行业标准" 转化为 "平台 + 模块 + 工艺流程 + 通量 + 现场条件" 的可交付配置方案。根据输入信息的确定性，提案天然分为两类：

类型 A国标/行标驱动型

特征：客户明确给出国标/行标编号，工艺流程和参数在标准文档中强制规定。

案例：成都食检院 — 硝基呋喃 + 127 种药物残留，依据 GB 31656.13、GBT 21311、GB 31658.26。

AI 的职责：精准解析标准 → 套工艺 → 选平台/模块 → 算通量 → 出文档。

难点：标准细节、参数一致性、合规性审核。

类型 B讨论驱动型

特征：无现成标准，工艺路线需要与客户来回讨论、迭代确认。

案例：中药经典名方质量分析 — 工艺由客户科研团队定义，时序图、并行优化由双方共创。

AI 的职责：结构化采集客户需求 → 拆解实验时序 → 推荐模块组合 → 画全流程图 → 迭代优化。

难点：需求钓稳、时序拆解、并行度优化、可视化表达。

两类提案共享的基础层

虽然表层流程不同，但两类提案在底层依赖同一组知识库与同一套系统抽象，仅在上层的"采集/分析/生成"环节分叉：

2.2 平台卡片（Platform Card）

每个平台用一份 Markdown + 结构化 frontmatter 描述。建议使用 YAML frontmatter 让 Claude 可精准抽取字段：

---

name: 标准型 1400 平台
dimensions_mm: { length: 1400, width: 850, height: 1200 }
max_modules: 6
max_trays: 8
tray_size_mm: { length: 300, width: 200 }
supported_functions: [称量, 提取, 浓缩, 净化, 浓缩定容, 过滤, 萃取, 定容, 消解]
pipette_options_ml: [10, 5, 2, 1]
gripper_options: [试管-15ml, 试管-50ml, 离心管-2ml]   # 待落实
power_kva_nominal: 3.0
power_kva_peak: 4.5

# 标准型 1400 平台

## 用途说明
最常用的主力平台，可承载 6 个模块 + 8 个托盘，兼顾通量和灵活性。

## 典型配置示例
- 提取平台：3× 加液振荡 + 1× 离心 + 1× 移液 + 1× 加内标
- 净化平台：3× SPE + 2× 浓缩 + 1× 定容

## 限制
- 单平台最大功耗 4.5 kVA，配 SPE + 浓缩时需确认电力余量
- 超过 4 个加热类模块时需强制通风

对三种平台（800、1200、1400）各建一份，共 3 份；未来新平台按相同模板追加。

2.3 模块卡片（Module Card）

公司 150+ 模块是提案的"原子零件"，必须结构化。建议字段：

基础信息

  • id：唯一 ID（如 MOD-CF-5K）
  • name：模块名称
  • category：离心/振荡/SPE/浓缩/加液/过滤/…
  • version：硬件版本

功能参数

  • function_params：结构化键值对（转速范围、温度、时间、管位数等）
  • capacity：单次处理量（如 24 管）
  • throughput_per_hour：单模块每小时产量

物理约束

  • footprint_mm：占用空间 L×W×H
  • weight_kg：重量
  • mount_platform_types：可部署到哪些平台

电力与特殊需求

  • power_w：额定功率
  • voltage：电压（220V/380V）
  • special_requirements：如通风/低振动/排废
  • compatible_modules：需要与哪些模块搭配
  • conflicts_with：与哪些模块互斥
实施建议：先从高频使用的 20~30 个核心模块入手建卡，一周内建完；剩余 120+ 模块用模板批量补。同时建一个 modules-index.md，列出所有模块 ID + 名称 + 分类，便于 Claude 快速检索。

2.4 工艺流程库（Process Template）

每个主流程一份 Markdown，包含：适用范围、依据标准、分步骤流程（含关键参数）、推荐模块组合、典型时序、基质差异化参数表。当前已建 process-quechers.md（植物源）、process-inorganic.md；需补动物源、茶叶、香辛料、中药等。

2.5 国标库（Standards Library）

每个国标/行标一节，至少包含：

  • 标准编号、标题、发布日期、代替关系
  • 适用范围（基质/分析物）
  • 关键前处理步骤（称样量、试剂、盐包、离心、净化、浓缩、定容）
  • 关键参数表（可被 AI 直接引用）
  • 后端检测仪器要求

2.6 后端仪器兼容库

列出常见 HPLC-MS/MS、GC-MS/MS、ICP-OES、ICP-MS、UPLC、LC-QTOF 的品牌/型号/进样规格/兼容性。前处理的定容体积、进样管规格由此反推。

2.7 历史样本库（samples/）

每个历史项目保留：原方案文件 + 一份 summary.md（摘要客户场景、选用配置、通量、成交价区间 [可选]、客户反馈）。样本摘要让 AI 做 few-shot 参考。

3.4 各阶段实现要点

阶段 ① — 新建项目

触发口令 新建提案 {客户名}，在 projects/{客户名}/ 下建目录，初始化 workflow-status.md。由 proposal-workflow Skill 完成，当前已落地。

阶段 ② — 标准解析

用户把客户给的 PDF 拷贝到 projects/{客户名}/_inputs/，然后说"解析标准"。Claude 会：

  1. 读每份 PDF，抽取标准编号、适用范围、关键参数
  2. 在 references/standards-library.md 中查找是否已有对应条目；有则复用，无则提示用户"我先按 PDF 生成草稿条目，请你人工校核后入库"
  3. 输出 projects/{客户名}/standards-extracted.md，列出每个标准的核心参数表
关键：PDF 抽参数不稳定。让 Claude 读，生成草稿条目；人工校对后才入库 — 这是保证"不编造"的最后一道护栏。
阶段 ③ — 信息采集

沿用 proposal-collector Skill，六大类问题：业务需求 / 技术规范 / 行业规范 / 效率要求 / 布局 / 基础设施。

类型 A 场景下，技术规范类的问题可以被"已解析的标准"自动填充：比如检测方法、国标编号、分析物清单，直接从阶段 ② 继承，不必再问。

阶段 ④ — 自动匹配（核心智能）

这一步是整个 Pipeline 的"脑"，由 proposal-analyzer Skill 扩展后完成。决策算法如下：

  • 审核：对照国标库、模块库、客户需求逐项打勾；有 ❌ 自动改 v2；有 ⚠️ 列人工确认清单

3.5 类型 A 的最佳实践清单

✅ DO

4.4 每阶段实现要点

① 需求钓稳 — 迭代式填表

新建 Skill proposal-rnd-collector（研发型采集），与现有 proposal-collector 区分。Claude 按 G1~G10 逐项引导：每轮最多 2~3 个问题，客户每轮答一批，Claude 更新 collected-info.md 并显示完成度，遇到模糊项给出候选答案让客户选而不是开放性提问。

② 时序建模 — 从 SOP 到 Gantt

新建 Skill proposal-sequence-builder，读 G4 步骤，生成如下 Gantt 时序图：

gantt title 中药经典名方前处理时序 dateFormat HH:mm axisFormat %H:%M section 工位A 称量 :a1, 00:00, 5m 加提取液 :a2, after a1, 3m 超声提取 :a3, after a2, 30m section 工位B 离心 :b1, after a3, 10m 过滤 :b2, after b1, 5m section 工位C 浓缩 :c1, after b2, 40m 定容 :c2, after c1, 5m

然后把 Gantt 再转成 Swimlane SVG，在 HTML 中交互展示 — 这正是 06-中药方案.html 的实现路径。

③ 模块-工位映射

每个步骤查询模块库候选模块 → 按时间占用找可并行的工位 → 合并成平台。关键计算：节拍 = max(各工位链路耗时)。

④ 并行优化

这是大连项目 V1→V2 的核心增量："离心时段并行准备西林瓶和过滤平台"。抽象为三种优化手法：

  • 批次错峰：在 A 批次进入离心时，B 批次开始称量
  • 工位预热：长耗时模块（如浓缩）前置准备
  • 夹爪复用：同夹爪类型的物料合并工位

输出 optimization-report.md，对比 V1 / V2 节拍提升率。

⑤ 可视化发布

大连项目 HTML 是质量标杆。把它做成模板 — 抽出 CSS / SVG 结构 / 交互脚本，放到 references/viz-templates/，下次 AI 只需要把内容填进模板即可。

4.5 类型 B 的最佳实践

✅ DO
  • 让客户先填 G1~G10 表格再开始生成 — 信息不够宁可暂停
  • 把 06-V1.2 那份 HTML 当作 视觉模板，保留其 CSS 和 SVG 结构
  • 时序图先 Mermaid Gantt 快速出，再升级为交互 SVG
  • 每轮迭代保留版本号：V1.0 / V1.1 / V1.2，客户可追溯
  • 在关键工位放"客户确认锚点"，避免 AI 推演过头
❌ DON'T
  • 不要让 AI 自行发明关键工艺参数（必须来自客户 SOP）
  • 不要一次性出最终 HTML — 至少拆 3 步：采集→时序→可视化
  • 不要用自由格式对话采集，一定要按 G1~G10 结构
